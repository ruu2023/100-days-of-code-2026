# CSV Import 複数明細取り込み 仕様

## 背景

現行の CSV Import は 1 回の解析につき 1 ファイルを前提とし、`CsvImport` 1 件ごとに 1 テーブルを新規作成する。

- 画面: `app/views/csv_imports/index.html.erb`
- 解析: `CsvImports::Analyzer`
- 取込: `CsvImports::Importer`
- 検索: `CsvImports::Searcher`

今回追加したい要件は、クレジットカード明細 CSV を数ヶ月分まとめて取り込み、同一形式のファイル群は 1 テーブルへ集約できるようにすること。

## 目的

- 複数の明細 CSV を 1 回の操作でまとめて登録できる
- 同一ヘッダー構成の CSV は同じ保存先テーブルへ取り込める
- カード会社ごとに形式が異なっても、形式単位で取り込み先を分けられる
- 既存の「取り込み済みテーブルを個別に見る」運用は維持する
- 将来的な横断検索に拡張しやすい構造にする

## スコープ

### 対象

- CSV Import 一覧画面の入力拡張
- 複数ファイル解析
- 同一ヘッダー判定
- テーブル作成または既存テーブルへの追記
- 取込履歴の管理
- 既存検索画面に影響を出さない範囲でのデータ管理見直し

### 非対象

- 異なるヘッダー構成を 1 テーブルへ統合する機能
- 自由 SQL 検索
- カード会社ごとの個別パーサー実装
- 自動カテゴリ分類や家計簿ロジック
- 取込済みデータの更新・削除・Upsert

## 用語

- 取込セッション: ユーザーが一度にアップロードする複数 CSV のまとまり
- 形式グループ: 正規化後のヘッダーが完全一致する CSV のグループ
- データセット: 形式グループに対応する物理テーブルとその定義
- 取込ファイル: 取込セッションに含まれる個々の CSV ファイル

## 現状課題

- `CsvImport` が「1ファイル = 1レコード = 1テーブル」の責務を持っている
- `source_file` が単一添付なので複数ファイルを扱えない
- `target_table_name` が毎回新規作成前提で、既存テーブルへの追記ができない
- 検索対象が `CsvImport` 単位のため、形式グループ単位の管理がない

## 要件

### 1. アップロード要件

- 一度に複数 CSV ファイルを選択できる
- 入力 UI は単一ファイル選択から複数ファイル選択へ変更する
- 表示名は取込セッション名として扱う
- 未入力時は日時ベース、または先頭ファイル名ベースで補完する

### 2. 解析要件

- ファイルごとに区切り文字、ヘッダー有無、行数、推定カラムを解析する
- ヘッダー比較には「正規化後ヘッダー配列」を使う
- 正規化ルールは現行 `Analyzer#normalize_headers` を踏襲する
- 正規化後ヘッダー配列が完全一致したファイル同士を同一形式グループとみなす
- 順序違いは別形式として扱う
- ヘッダーなし CSV は対象外とするか、V1 では取り込み不可にする

### 3. 保存先判定要件

- 同一取込セッション内で同じヘッダーのファイルは 1 データセットに集約する
- 既存データセットとヘッダーが一致する場合は、その既存テーブルへ追記できる
- 一致する既存データセットがない場合のみ新規テーブルを作成する
- 「同一ヘッダーのみ同じテーブル」という判定を厳格に守る

### 4. 取込要件

- 取込実行時は形式グループごとにまとめて insert する
- 同じグループ内の複数ファイルは同一テーブルへ連続投入する
- 取り込み順はファイル選択順、またはファイル名昇順で固定する
- ファイル単位の成功・失敗結果を保持する
- 1 ファイル失敗時の扱いは V1 では「グループ単位でロールバック」を推奨する

### 5. 閲覧要件

- 現状どおり、取り込み済みテーブルを個別に開いて確認できる
- 一覧画面では取込セッションとデータセットの関係が分かること
- どのファイルがどのテーブルに入ったか追跡できること

### 6. 検索要件

- 当面は現状の個別テーブル検索を維持する
- 将来の横断検索に備え、どのデータセットが検索対象か識別できる管理情報を保持する
- 横断検索は V2 とし、今回の必須実装からは外してよい

## 推奨仕様

### UI 仕様

`csv_imports/index` の新規入力欄を以下に変更する。

- 表示名
- CSV ファイル複数選択
- 解析ボタン

補助表示:

- 複数選択可能である説明
- 同一ヘッダーは同じテーブルへまとまる説明
- 異なる形式は別テーブルになる説明

一覧表示の推奨項目:

- 取込セッション名
- ファイル数
- 生成または追記されたデータセット数
- ステータス
- 作成日時

### データモデル仕様

現行 `csv_imports` をセッション管理へ寄せ、ファイル管理とデータセット管理を分離する。

#### `csv_imports`

取込セッション管理。

- `name`
- `status`
- `error_message`
- `imported_at`
- `files_count`
- `datasets_count`

#### `csv_import_files`

アップロードされた個別ファイル管理。

- `csv_import_id`
- `source_filename`
- `storage attachment`
- `col_sep`
- `header_row`
- `row_count`
- `normalized_headers` JSON
- `inferred_columns` JSON
- `sample_rows` JSON
- `status`
- `error_message`
- `csv_import_dataset_id`

#### `csv_import_datasets`

ヘッダー単位の保存先管理。

- `name`
- `header_signature`
- `normalized_headers` JSON
- `inferred_columns` JSON
- `target_table_name`
- `status`
- `row_count`
- `source_kind`

補足:

- `header_signature` は正規化後ヘッダー配列を join した文字列、または digest
- `target_table_name` はデータセット単位でユニーク
- 既存検索画面は `csv_import_dataset` を軸に移すと整理しやすい

### テーブル振り分け仕様

1. 各ファイルを解析する
2. `normalized_headers` を作る
3. `header_signature` を作る
4. 同一セッション内で `header_signature` ごとにグルーピングする
5. 既存 `csv_import_datasets.header_signature` を検索する
6. 一致があれば既存テーブルへ追記する
7. なければ新規テーブルを作る

## バリデーション

- ファイルが 1 件以上必要
- CSV 以外は拒否
- ヘッダーなしは V1 では拒否を推奨
- 同一ファイル名の重複は許可。ただし内容重複チェックは V1 では行わない
- 同一グループ内でも列数が一致しないファイルはエラー
- 既存データセットへ追記する場合、`inferred_columns` と型定義が一致しない場合はエラー

## エラーハンドリング

- 解析失敗はファイル単位で記録する
- 取込失敗はデータセット単位で記録する
- 画面には以下を表示する
  - 失敗したファイル名
  - 失敗理由
  - 取り込まれなかった対象テーブル
- V1 は部分成功よりも「何が入ったかが明確」な挙動を優先する

## 検索の将来拡張

横断検索を見据えるなら、以下のメタ情報をデータ行に追加できる設計が望ましい。

- `import_dataset_id`
- `import_file_id`
- `source_filename`
- `imported_at`

ただし既存の動的テーブルへ列追加が必要になるため、V1 では必須にしない。
V2 で以下のいずれかを検討する。

- 各動的テーブルへ共通メタ列を追加する
- 横断検索用のインデックステーブルを別に持つ
- PostgreSQL の `UNION ALL` ビューを生成する

## 画面遷移

### 一覧画面

1. ユーザーが複数 CSV を選択
2. 解析実行
3. 解析結果画面へ遷移

### 解析結果画面

- 取込セッション概要
- 形式グループ一覧
- 各グループに含まれるファイル一覧
- 保存先:
  - 新規テーブル作成
  - 既存テーブルへ追記
- 実行ボタン

### 詳細画面

- データセット単位でプレビュー
- 個別テーブル検索
- どのファイルを取り込んだかの履歴表示

## V1 実装方針

段階的に進める。

### Step 1

- `CsvImport` を取込セッションへ寄せる
- `CsvImportFile` と `CsvImportDataset` を追加する
- 複数ファイル解析 UI を追加する

### Step 2

- ヘッダー一致でグルーピングする
- 新規テーブル作成または既存テーブル追記を実装する
- 解析結果画面にグループ単位の保存先表示を追加する

### Step 3

- 詳細画面をデータセット中心へ変更する
- ファイル履歴表示を追加する
- 既存検索機能をデータセットへ接続し直す

### Step 4

- 横断検索の PoC を追加する

## 受け入れ条件

- 3 ファイル以上を一括選択して解析できる
- 同じカード会社・同じヘッダーの複数 CSV が 1 テーブルへ入る
- 異なるカード会社でヘッダーが異なる CSV は別テーブルになる
- 既存と同じヘッダーなら既存テーブルへ追記できる
- どのファイルがどのテーブルへ入ったか画面から確認できる
- 既存の個別テーブル検索が継続利用できる

## 補足判断

- 「テーブル検索は横断してできるようにしたいが、現状のままでも一旦 import 済みのテーブルが見れれば OK」という前提から、今回の仕様では横断検索を V2 扱いに落とす
- まずは「複数ファイルを安全に同一テーブルへ取り込める」ことを最優先にする
- 厳密一致のヘッダー判定にしておくと、カード会社差分による事故を避けやすい
